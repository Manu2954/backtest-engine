                     CRITICAL BUG FIX: MULTI-COLUMN INDICATORS
================================================================================
Implementation Date: 2026-03-15
Status: FIXED ✅

PROBLEM:
Backtests were failing with "ValueError: Operand column not found: macd_2"
when users referenced multi-column indicators by their base alias in conditions.

ROOT CAUSE:
Multi-column indicators (MACD, BB, STOCH, ICHIMOKU) create sub-columns with
suffixes but NOT the base alias:
  - MACD "macd_2" creates: macd_2_macd, macd_2_signal, macd_2_hist (no "macd_2")
  - BB "bb_20" creates: bb_20_upper, bb_20_mid, bb_20_lower (no "bb_20")

SOLUTION:
Modified backend condition_engine.py to automatically map base aliases to
their primary sub-column:
  - macd_X → macd_X_macd
  - bb_X → bb_X_mid
  - stoch_X → stoch_X_k
  - ichi_X → ichi_X_tenkan

FILES MODIFIED:
  - backend/app/engine/condition_engine.py (mapping logic in 2 functions)
  - backend/tests/unit/test_multicolumn_indicator_mapping.py (NEW - 10 tests)
  - backend/tests/unit/test_condition_engine.py (minor fixes)

TESTING:
✅ All 10 new mapping tests pass
✅ All 24 existing condition engine tests pass
✅ Backward compatible with existing strategies

For full details, see: docs/BUG_FIX_MULTI_COLUMN_INDICATOR_MAPPING.txt
