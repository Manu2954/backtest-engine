================================================================================
           BUG FIX: BENCHMARK CALCULATION WITH WARMUP PERIOD
================================================================================
Date: 2026-03-15
Priority: HIGH
Status: FIXED

================================================================================
                                THE PROBLEM
================================================================================

SYMPTOM:
When running backtests for ^NSEI (and other tickers), the benchmark (buy-and-hold)
results were incorrect compared to AAPL. The benchmark was showing different
returns than expected for the same date range.

USER REPORT:
"when I ran a strategy for ^NSEI ticker with proper start date and end date as
AAPL i got the proper Benchmark results for AAPL but not for ^NSEI"

ROOT CAUSE:
The benchmark was being calculated AFTER the indicator warmup period was trimmed.

DETAILED EXPLANATION:
1. User requests backtest from 2020-01-01 to 2023-12-31 (full year)
2. System fetches OHLCV data for full range (365 bars)
3. System computes indicators (e.g., SMA 50 requires 50 bars warmup)
4. System trims warmup period - removes first 50 bars where indicators are NaN
5. Strategy backtest runs from bar 50 to bar 365 (315 bars)
6. ❌ PROBLEM: Benchmark was calculated from bar 50 to bar 365 (using trimmed data)
7. This means benchmark buys at a different price than expected

EXAMPLE:
- User wants to test Jan 1 to Dec 31
- After warmup trim, strategy starts from Feb 19 (50 days later)
- Benchmark was buying at Feb 19 prices instead of Jan 1 prices
- For ^NSEI, if price moved significantly in those 50 days, benchmark is wrong

CODE LOCATION:
backend/app/tasks/backtest_task.py, lines 70-134:
```python
df = compute_indicators(df, indicators)  # Line 70
df, warmup_bars = trim_warmup_period(df)  # Line 74 - trims data
...
# Line 132 - calculated benchmark from TRIMMED data
benchmark_equity = calculate_buy_and_hold_equity(
    df, float(run.initial_capital), asset_class=run.asset_class
)
```

backend/app/engine/report_generator.py, lines 112-173:
The benchmark stats calculation didn't align benchmark to strategy start date,
so it compared full benchmark returns against partial strategy returns.

================================================================================
                                THE SOLUTION
================================================================================

TWO-PART FIX:

1. PRESERVE ORIGINAL OHLCV DATA (backend/app/tasks/backtest_task.py)
   Store the original OHLCV data BEFORE warmup trim, use it for benchmark:

   ```python
   # Store original OHLCV data for benchmark calculation (before warmup trim)
   df_original = df[["open", "high", "low", "close", "volume"]].copy()

   df = compute_indicators(df, indicators)
   df, warmup_bars = trim_warmup_period(df)

   # Calculate benchmark from ORIGINAL data (not trimmed)
   benchmark_equity = calculate_buy_and_hold_equity(
       df_original, float(run.initial_capital), asset_class=run.asset_class
   )
   ```

2. ALIGN BENCHMARK TO STRATEGY DATE RANGE (backend/app/engine/report_generator.py)
   When calculating benchmark stats, slice benchmark to match strategy start date:

   ```python
   # IMPORTANT: Align benchmark to strategy's date range
   strategy_start = strategy_equity.index[0]
   if strategy_start in benchmark_equity.index:
       benchmark_equity = benchmark_equity.loc[strategy_start:]
   else:
       benchmark_equity = benchmark_equity[benchmark_equity.index >= strategy_start]

   # Get benchmark initial capital at strategy start date
   benchmark_initial = float(benchmark_equity.iloc[0])

   # Calculate returns from aligned period
   benchmark_return_pct = _safe_div(
       benchmark_final - benchmark_initial, benchmark_initial
   ) * 100
   ```

================================================================================
                            HOW IT WORKS NOW
================================================================================

CORRECT FLOW:

1. User requests backtest: 2020-01-01 to 2023-12-31
2. Fetch OHLCV: 365 bars from Jan 1 to Dec 31
3. Store original: df_original = copy of OHLCV (365 bars)
4. Compute indicators on df
5. Trim warmup: df now has 315 bars (starting from Feb 19)
6. Run backtest: Uses df (315 bars, Feb 19 to Dec 31)
7. Calculate benchmark: Uses df_original (365 bars, Jan 1 to Dec 31)
8. Generate report: Aligns benchmark to strategy start date (Feb 19)

ALIGNMENT IN REPORT:
- Strategy equity: Feb 19 to Dec 31 (initial capital = $10,000)
- Benchmark equity: Feb 19 to Dec 31 (initial capital = Feb 19 value)
- Strategy return: (final - $10,000) / $10,000
- Benchmark return: (final - Feb 19 value) / Feb 19 value
- Alpha: Strategy return - Benchmark return ✅ Fair comparison!

WHY THIS IS CORRECT:
The benchmark represents "what if you bought and held from the ACTUAL start
date (Jan 1)?" but we compare returns from when the strategy STARTED trading
(Feb 19). This is fair because:
- Strategy can't trade during warmup period (indicators not ready)
- Benchmark should reflect holding from actual start (Jan 1)
- But returns comparison should be from when strategy became active (Feb 19)

================================================================================
                            KNOWN EDGE CASES
================================================================================

1. VERY LARGE WARMUP PERIODS:
   If warmup period consumes most of the data (e.g., 200-day SMA on 220-day range),
   the strategy only runs for 20 bars, but benchmark comparison is still valid
   because it's aligned to when strategy started trading.

2. MULTIPLE INDICATORS WITH DIFFERENT WARMUP:
   The system uses the longest warmup period (trim_warmup_period handles this).
   Benchmark aligns to when ALL indicators are ready.

3. NO INDICATORS:
   If strategy has no indicators, no warmup trim occurs, benchmark starts from
   day 1 as expected. No change in behavior.

4. INTRADAY DATA:
   Works the same way - if 1-hour bars need 50-bar warmup, benchmark aligns
   to 50 hours after start.
