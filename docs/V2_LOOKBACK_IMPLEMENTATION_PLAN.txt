# V2 Feature: Lookback Comparisons - Implementation Plan

Generated: 2026-03-08
Feature: Enable comparing current values to historical values (N bars ago)

---

## GOAL

Allow conditions like:
- "ADX > ADX from 3 bars ago" (ADX is rising)
- "Close > Close from 10 bars ago" (price momentum)
- "Chikou > price from 26 bars ago" (Ichimoku confirmation)
- "RSI < RSI from 5 bars ago" (RSI is falling)

---

## DESIGN OPTIONS

### Option 1: Add IS_RISING_N / IS_FALLING_N Operators
```json
{
  "left_operand_type": "INDICATOR",
  "left_operand_value": "adx",
  "operator": "IS_RISING_N",
  "right_operand_type": "SCALAR",
  "right_operand_value": "3"  // Lookback period
}
```

**Pros:**
- Simple API for common use case
- Easy to understand: "ADX is rising over 3 periods"
- No new operand types needed

**Cons:**
- Limited to rising/falling checks
- Can't do complex comparisons (e.g., "close > close[-10] + 5")
- Need separate operators for each trend direction

### Option 2: Add LOOKBACK Operand Type
```json
{
  "left_operand_type": "INDICATOR",
  "left_operand_value": "adx",
  "operator": "GT",
  "right_operand_type": "LOOKBACK",
  "right_operand_value": "adx:-3"  // Format: "column:offset"
}
```

**Pros:**
- Flexible: Works with any operator (GT, LT, CROSSES_ABOVE, etc.)
- Can compare any column to its historical value
- Clean separation of concerns

**Cons:**
- Requires string parsing: "adx:-3"
- More complex API
- Users need to understand offset notation

### Option 3: Extend Operand Value with Syntax
```json
{
  "left_operand_type": "INDICATOR",
  "left_operand_value": "adx",
  "operator": "GT",
  "right_operand_type": "INDICATOR",
  "right_operand_value": "adx[-3]"  // Pandas-like syntax
}
```

**Pros:**
- Intuitive syntax (matches pandas)
- No new operand types
- Works with existing operators

**Cons:**
- Requires parsing square brackets
- Overloads INDICATOR type (now has two behaviors)
- Error-prone (what if column name has brackets?)

---

## RECOMMENDED APPROACH: Option 2 (LOOKBACK Operand Type)

**Rationale:**
- Most flexible and explicit
- Clean separation from existing types
- Easy to validate and parse
- Supports future enhancements (forward lookback for Ichimoku)

---

## IMPLEMENTATION DETAILS

### 1. Update Condition Schema

**New operand type:**
- `LOOKBACK` - Reference to a column value N bars ago

**Format:**
- `"column_name:-N"` for N bars ago
- `"column_name:+N"` for N bars forward (future feature, for Ichimoku cloud)

**Examples:**
- `"adx:-3"` = ADX from 3 bars ago
- `"close:-26"` = Close from 26 bars ago
- `"ichi_span_a:+26"` = Span A 26 bars forward (future)

### 2. Modify condition_engine.py

**Add to _resolve_operand():**
```python
elif op_type == "LOOKBACK":
    # Parse "column:offset" format
    parts = op_value.split(":")
    if len(parts) != 2:
        raise ValueError(f"Invalid LOOKBACK format: {op_value}. Expected 'column:offset'")

    column_name = parts[0]
    try:
        offset = int(parts[1])
    except ValueError:
        raise ValueError(f"Invalid offset in LOOKBACK: {parts[1]}. Must be integer.")

    if column_name not in df.columns:
        raise ValueError(f"Column not found: {column_name}")

    # Shift the series by the offset
    # Negative offset = look back (shift forward in index)
    # Positive offset = look ahead (shift backward in index)
    return df[column_name].shift(-offset)
```

**Example:**
- `shift(-offset)` when offset=-3 becomes `shift(3)` = look back 3 bars
- `shift(-offset)` when offset=+3 becomes `shift(-3)` = look ahead 3 bars

### 3. Handle Edge Cases

**NaN values at boundaries:**
- Lookback N bars: First N bars will be NaN
- Lookahead N bars: Last N bars will be NaN
- Crossover operators already handle NaN via `fillna(False)`

**Validation:**
- Ensure offset is within reasonable bounds (-1000 to +1000)
- Validate column exists before shifting
- Clear error messages for parsing failures

## FUTURE ENHANCEMENTS

Once LOOKBACK is stable:

1. **Forward lookback (lookahead):**
   - Format: "column:+N" for N bars ahead
   - Use case: Ichimoku future cloud validation
   - Warning: Can introduce look-ahead bias in backtests

2. **Relative lookback:**
   - IS_RISING_N, IS_FALLING_N operators
   - Syntactic sugar over LOOKBACK + GT/LT
   - Easier to understand for common case

3. **Lookback with offset ranges:**
   - "highest in last 20 bars"
   - "lowest in last 10 bars"
   - Requires aggregation, not just shift

## RISKS & MITIGATIONS

**Risk 1: Performance impact from multiple shifts**
- Mitigation: Pandas shift() is optimized, minimal overhead
- Benchmark: Test with 10+ lookback conditions on 10k bars

**Risk 2: User confusion with offset notation**
- Mitigation: Clear documentation and examples
- Add validation with helpful error messages
- Consider adding IS_RISING_N operator as alias

**Risk 3: Look-ahead bias with positive offsets**
- Mitigation: Document clearly that +N is for visualization only
- Add warning when using positive offset in backtest
- Consider blocking positive offsets by default

**Risk 4: NaN propagation at boundaries**
- Mitigation: Already handled by fillna(False) in crossover operators
- Document that first N bars won't trigger conditions with lookback N
- This is expected behavior
